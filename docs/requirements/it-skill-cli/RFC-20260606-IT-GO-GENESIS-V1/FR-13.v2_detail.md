# Functional Requirement: itgo Sovereign Manager (FR-13.v2 - Refined)

## 1. Overview
เปลี่ยนให้ `itgo` เป็นศูนย์กลางการจัดการทักษะ (Sovereign Manager) เพียงหนึ่งเดียว แทนที่คำสั่ง `go` และ `install` เดิม โดยเน้นความถูกต้องของข้อมูลเวอร์ชันที่ต้องอ้างอิงจากต้นฉบับจริง (Source-Aware)

## 2. Scope Consensus (The 4-Layer Logic)

### 2.1 Requirement Mapping (AI Synthesis)
- ยุติการทำงานของคำสั่ง `go` (Legacy Decommissioning)
- ขยายความสามารถของ `itgo` ให้รับ Argument เป็นชื่อโปรไฟล์ (Sovereign Installation)
- **Source-Aware Rich Manifest**: ระบบดัชนีต้องเก็บเลขเวอร์ชันที่สกัดจากไฟล์ `SKILL.md` ต้นทางจริง ไม่ใช่ใช้เวอร์ชันของตัวติดตั้ง (Installer) เพียงอย่างเดียว

### 2.2 Information Gathering (Research)
- **IG-1 (Technical)**: ใช้ข้อมูล `version` ที่สกัดได้จากระบบสแกนส่วนกลาง (CR-001 Refined Discovery) มาบันทึกลงใน `.it-skill.json`
- **IG-2 (Operational)**: เพื่อให้ Dashboard แสดงข้อมูลที่ถูกต้องสำหรับ Hybrid Fleet (ทักษะ [IT] ผสมกับ [A])

### 2.3 Implementation Governance (Layer 4)
- **Execution Skill**: `build-patch`

### 2.4 Pathway (Confirmation)
- **Confirm Command**: **`brfc-Phase 2 confirm`**

## 3. ตรรกะการทำงานหลัก (Core Logic)
1. **Command Centralization**: `itgo` ทำหน้าที่เป็น Gateway เดียวสำหรับการติดตั้งและเรียกดูสถานะ
2. **Accurate Indexing**: ในขั้นตอนการติดตั้ง ให้ดึงค่า `version` จาก Object `Skill` (ที่สแกนมาแล้ว) บันทึกลงในไฟล์ดัชนี
3. **Hybrid Manifest**: รองรับการเก็บข้อมูลเลขเวอร์ชันที่หลากหลาย (e.g., v1.1.0 สำหรับงาน IT และ v26.4.18 สำหรับงาน Arra)

## 4. มาตรฐานการแสดงผล (Output Standard)
- Dashboard ต้องแสดงเลขเวอร์ชันที่ตรงกับไฟล์จริงในเครื่อง โดยอ่านผ่านไฟล์ดัชนี JSON เพื่อความรวดเร็ว
